$是干嘛的？
vue一些有用的实例属性与方法，和用户定义的属性区分开来。

一、router 和 route的区别？

$router对象是全局路由的实例，
.push()跳转页面（和等价），会向history栈添加一个新的记录，返回上一页可以看到之前的页面。
.go()页面路由跳转，前进或后退，.go(-1)返回上一页
.replace()替换当前页面，不会入history栈。

$route对象是当前路由。包括当前路径，参数，query对象等。

所以一般传参的时候会看到，a页面this.$router.push(…query:{..})，b页面接收this.$route.query.id

二、query和params传参区别：
query刷新页面，参数不会消失，而params刷新页面，参数会消失。
query的参数会显示在地址栏，params不会。
query配合name 或 path都可以，params只能配合name使用。

因为刷新消失，params传参时，router-index.js中路由的path要改成“/login/:id”动态路由的形式，这样就会把接收到的参数替换为地址，比如 params:{id: 3},那么最终访问的地址是…/login/3

三、SPA
SPA：
单页应用，通过重写当前页面来与用户交互，避免页面之间的切换打断用户体验。
在单页应用中，根据需要动态加载适当的资源。
MPA：
多页应用，每个页面都是独立的，当访问另一个页面时，需要重新加载html，css，js文件。
区别：
SPA：一个主页面和多个页面片段 => 局部刷新 => 页面切换速度快，用户体验好 => 数据传递容易 => 难实现SEO搜索引擎优化，可用SSR实现

MPA：多个主页面 => 整页刷新 => 页面切换加载资源，速度慢 => 通过url、localstorage、cookie等数据传递 => 容易实现SEO

但SPA首次渲染速度相对较慢。

四、SPA的两种实现：
SPA的核心是 前端路由。
前端路由的核心是 改变视图的同时不会向后端发送请求。

前端路由：通过js代码，处理不同url和不同组件模块的对应关系。
这些url在后端获取的都是整个页面资源，而在浏览器端，js通过判断不同的url，渲染不同的组件模块。
因此：SPA 切换组件不刷新页面；url与组件一一对应；浏览器的前进后退按钮有效。

hash模式和history API模式
hash：通过url中的hash部分（#后的锚点值）来映射不同的组件，改变hash，也不会向服务器重新请求页面，也不会刷新页面，但会向浏览器的历史记录栈压入新的记录，从而实现浏览器的前进后退按钮。
history API：通过history.replaceState 和 history.pushState改变历史记录和url

补充：什么是SSR？
服务端渲染，在服务端起一个node应用，请求发送过来的时候，先拦截异步js请求，提前将数据填充到html页面中返回浏览器。这样爬虫抓到的页面就是带数据的。vue用Nuxt.js框架。

SPA（单页应用）首屏加载速度慢怎么解决？

为什么首屏加载速度慢？
首屏时间：浏览器从响应用户输入网址，到首屏内容渲染完成 的时间。
网络延时、资源文件体积大、有重复发送请求、加载脚本时，渲染内容堵塞。

如何优化首次渲染速度慢？
主要包括两部分，资源加载 的优化 和 页面渲染 的优化
针对资源文件体积大：
（1）可以减小入口文件体积，比如利用懒加载，把不同路由对应的组件分割成不同的代码块，被请求时单独打包。vue-router配置的时候，动态加载路由，以函数的形式加载路由，只有解析到相应路由时，加载相应组件。

routes:[
    path: 'xx',
    name: 'xx',
    component: () => import('...')
]

什么是懒加载？
延迟加载一些非关键资源，按需加载。

（2）对于减少http请求次数：静态资源本地缓存：对于后端返回的资源，采用http缓存，或Service Worker离线缓存。前端合理利用localStorage。

五、vue-router的两种路由模式：
vue为了构建SPA单页面应用，引入前端路由系统，其核心在于，改变视图的同时不会向后端发出请求。
hash模式：#符号，比如 http://www.abc.com/#/hello， hash的值为 #/hello。hash虽然出现在url中，但不会被包含在http请求中，改变hash不会重新加载页面。

history模式：可以对历史记录栈进行修改，执行修改时，虽然改变了当前url，但不会立即向后端发送请求。

选择mode类型后，程序会根据不同的mode类型创建不同的history对象：
hash -> HashHistory：push和replace方法，push将新的路由添加到浏览器访问历史的栈顶，replace替换当前栈顶的路由。
因为hash发生变化的url都会被浏览器记录（历史访问栈）下来，从而你会发现浏览器的前进后退都可以用了。这样一来，尽管浏览器没有请求服务器，但是页面状态和url一一关联起来。

history -> HTML5History：不止可以读取历史记录栈，还可以修改：pushState() 和 replaceState()。back,forward,go切换页面

abstract -> AbstractHistory

实现：
hash：
利用了window.onhashchange事件，当hash改变，自动调用hashchange的监听事件，得到改变后的url。
history：
通过history API的pushState和replaceState方法，替换url而不刷新页面。不会向后端发送请求，但一旦刷新，就会真正发送请求。

history的优势：
pushState设置的url可以是同源的任意url，而hash只能修改#后面的部分；
pushState设置的新url可以与当前url一模一样，而hash必须不一样才会把记录添加到记录栈中。

history的缺点：刷新的时候，浏览器会向服务器发送请求，hash模式下仅#前的内容会包含在请求中，因此即便后端没有对url全覆盖，也不会轻易返回404，但history，就是当前url发送请求，如果后端没有对该url的路由处理，会返回错误。
解决：可以在服务端增加一个覆盖所有情况的候选资源，如果出现没配置的url，返回同一个index.html页面，避免404不好看。

总结：
vue => SPA => 前端路由 => url和组件一一对应 => hash模式和history API模式

hash：通过url中的hash部分（#后的锚点值）来映射不同的组件，不会发送http请求但会入栈，因此能实现浏览器的前进后退。通过hashchange监听，获取改变后的url。如果刷新，发送请求只有hash前的部分，因此不会轻易返回404.

history API：通过pushState和replaceState方法，可以对历史记录栈进行修改，虽然改变了当前url，但不会立刻向后端发送请求。需要和后端的配合，当刷新的时候，会发送请求，地址栏的url，因此容易404.要么前后端配置好，前端的url和后端的路由处理一致，要么将不存在的路径都重定向到某个页面。
