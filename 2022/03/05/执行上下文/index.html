执行上下文

2022-03-05

一、词法作用域（静态作用域）和动态作用域：

JavaScript深入之词法作用域和动态作用域

词法作用域：函数的作用域在函数定义的时候就决定了。

动态作用域：在函数调用的时候才决定。

举个例子：

var value = 1;
function foo(){
    console.log(value);
}
function bar(){
    var value = 2;
    foo();
}
bar()  //1

词法作用域的过程：

执行foo函数，先从foo函数内部查找是否有局部变量value，如果没有，就根据书写的位置，查找上一层的代码，得到value=1，结果打印1。

动态作用域的过程：

执行foo函数，先从foo函数内部查找是否有局部变量value，如果没有，就从调用函数的作用域，也就是 bar 函数内部查找 value 变量，所以结果会打印 2。

js是词法作用域，所以打印1。

二、执行上下文栈：

JavaScript深入之执行上下文栈

js引擎并非一行一行地分析和执行程序，而是一段一段地分析执行。当执行一段代码时，会进行一个“准备工作”（执行上下文）（这个工作不仅包含了预编译阶段的‘变量提升、函数提升’等，还包含了执行阶段）。

那么，这个一段一段的段是如何划分的？

首先说可执行代码：包括三种：全局代码、函数代码、eval代码。

然后，执行上下文栈：用来管理执行上下文

当js要开始解释执行代码的时候，首先遇到全局代码，向执行上下文栈push一个全局执行上下文（globalContext）。只有整个应用程序结束时，栈才会被清空，因此程序结束前，栈底永远有个globalContext。

举个例子：

function fun3(){
    console.log('fun3');
}
function fun2(){
    fun3();
}
function fun1(){
    fun2();
}
fun1();

当执行一个函数时，就会创建一个执行上下文，并压入执行上下文栈，函数执行完毕时，将其从栈中弹出。

// 伪代码
// fun1()
ECStack.push(<fun1> functionContext);
// fun1中竟然调用了fun2，还要创建fun2的执行上下文
ECStack.push(<fun2> functionContext);
// fun2还调用了fun3！
ECStack.push(<fun3> functionContext);
// fun3执行完毕
ECStack.pop();
// fun2执行完毕
ECStack.pop();
// fun1执行完毕
ECStack.pop();
// javascript接着执行下面的代码，但是ECStack底层永远有个globalContext

三、执行上下文：

当js执行一段可执行代码时，会创建对应的执行上下文

每个执行上下文，包括三个重要属性：

变量对象
作用域链
this

（一）变量对象(Variable object，VO)：

JavaScript深入之变量对象

存储了执行上下文中定义的变量和函数声明。

（1）全局上下文：的变量对象：

全局对象，一个Object构造函数实例化的对象。浏览器中，全局对象就是window对象。

（2）函数上下文：

活动对象(activation object, AO)。进入到执行上下文，才会被激活，其上的各种属性才能被访问。

活动对象和变量对象其实是一个东西，只是变量对象是规范上的或者说是引擎实现上的，不可在 JavaScript 环境中访问，只有到当进入一个执行上下文中，这个执行上下文的变量对象才会被激活，所以才叫 activation object ，而只有被激活的变量对象，也就是活动对象上的各种属性才能被访问。

是在进入函数上下文时刻被创建的。

执行过程：

两阶段：分析（进入执行上下文）和执行（代码执行）。

（1）进入执行上下文：

此时还没有执行代码，变量对象包括：

1.函数的形参（函数上下文），属性值为undefined
2.函数声明。函数的声明直接提升到作用域的最顶部。
3.变量声明

关于声明提升：

js引擎在上下文的创建阶段，或者说对js代码预处理的时候，会把函数和变量的声明提升到靠前的位置。（注：只提升了声明，由于赋值是在第二阶段也就是执行阶段做的，所以赋值没有提升。）

函数声明先于变量声明，如果同名，变量声明不会覆盖之。

函数声明，同名函数，后者覆盖前者。

函数表达式 var foo = function(){} 不存在函数提升的情况。

只有var，let和const词法环境，块级作用域，暂时性死区（不能在初始化之前，使用变量）。

JS变量声明提升

举个例子：

function foo(a){
    var b = 2;
    function c(){}
    var d = function(){}

    b = 3
}
foo(1)

在进入执行上下文后，这时候的 AO 是：

AO = {
    arguments: {
        0: 1,
        length: 1
    },
    a: 1,
    b: undefined,
    c: reference to function c(){},
    d: undefined
}

（2）代码执行

顺序执行代码，修改变量对象的值。

当代码执行完后，这时候的 AO 是：

AO = {
    arguments: {
        0: 1,
        length: 1
    },
    a: 1,
    b: 3,
    c: reference to function c(){},
    d: reference to FunctionExpression "d"
}

总结：

全局上下文的变量对象初始化是全局对象

函数上下文的变量对象初始化只包括 Arguments 对象

在进入执行上下文时会给变量对象添加形参、函数声明、变量声明等初始的属性值

在代码执行阶段，会再次修改变量对象的属性值

一些问题：

1.执行上下文的生命周期：

（1）创建阶段
在这个阶段中，执行上下文会分别创建变量对象，建立作用域链，以及确定this的指向。

（2）代码执行阶段
创建完成之后，就会开始执行代码，这个时候，会完成变量赋值，函数引用，以及执行其他代码。

都没有错，博主讲的主要是针对变量对象，而变量对象的创建是在EC(执行上下文)的创建阶段，所以侧重点主要是EC的生命周期的第一个阶段。

2.VO 和 AO 到底是什么关系？

未进入执行阶段之前，变量对象（VO）中的属性都不能访问，但进入执行阶段后，变量对象(VO)转变为了活动对象(AO)，里面的属性都能被访问了，然后开始进行执行阶段的操作。

它们其实都是同一个对象，只是处于执行上下文的不同生命周期。

注：

AO/VO 属于 ES3，在ES5 使用 词法环境(lexical environment) 来管理静态作用域。

词法环境，就是描述环境的对象，包括：

环境记录(Environment Record)：记录相应环境中的形参，函数声明，变量声明等。
对外部环境的引用(out reference)

首先，环境初始化：在执行函数/全局代码之前进行环境初始化；

设置outer reference：先确定当前环境的外部引用(函数环境与该函数的scope属性有关)
环境记录初始化：就是我们常遇到的声明提升

然后再执行环境中的代码。

JS的词法环境以及闭包

（二）作用域链：

JavaScript深入之作用域链

首先，在js中，作用域分为 全局作用域 ， 函数作用域 和 ES6的块级作用域。

全局作用域：代码在程序的任何地方都能被访问。
函数作用域：在固定的代码片段才能被访问。
块级作用域：{}

作用域可以用来隔离变量，不同作用域内的同名变量不会冲突。

变量取值，会到创建这个变量的函数的作用域中取值。

当查找变量的时候，会先从当前上下文的变量对象中查找，如果没有找到，就会从父级(词法层面上的父级)执行上下文的变量对象中查找，一直找到全局上下文的变量对象，也就是全局对象。这样由多个执行上下文的变量对象构成的链表就叫做作用域链。

注：作用域 vs 上下文：

函数作用域，在函数定义的时候就确定好了，参考前面的词法作用域。

函数执行上下文，其中this指向，由函数运行的时候决定的。

作用域代表一块范围，其作用是让变量在此范围内起作用，而执行上下文用来管理这些变量，包括代码执行前的准备工作、代码执行。

作用域只是一个“地盘”，其中没有变量。变量是通过作用域对应的执行上下文环境中的变量对象来实现的。函数每调用一次，都会产生一个新的执行上下文环境。因为不同的调用可能就有不同的参数。

所以：作用域中变量的值是在执行过程中确定的，而作用域是在函数创建时就确定的。如果要查找一个作用域下某个变量的值，就需要找到这个作用域对应的执行上下文环境，再在其中找到变量的值。

执行上下文和作用域的理解

（三）this指向：

JavaScript 的 this 原理

一个比较基础的例子：

var obj = {
    foo: function(){
        console.log(this.bar);
    }
    bar: 1;
}
var foo = obj.foo;
var bar = 2;

obj.foo()  //1
foo()  //2

最基本的原因，obj.foo()在obj环境执行，而foo()在全局环境执行。

为什么？

首先看内存的数据结构。

1.内存的数据结构

var obj = { foo: 5};

将一个对象赋值给变量obj：
js引擎会先在内存生成一个对象 {foo: 5}，然后将这个对象的内存地址赋值给变量obj。换句话说，变量obj是一个地址（reference），后面读取obj.foo，（1）首先从obj拿到内存地址，（2）然后从地址读出对象，（3）返回其foo属性。

原始的对象以字典的结构保存，每一个属性对应一个属性描述对象。比如foo属性，对应一个属性描述对象，其中foo属性的值5，存在属性描述对象的value属性里面。

{
    foo: {
        [[value]]: 5
        [[writable]]: true
        [[enumerable]]: true
        [[configurable]]: true
    }
}

注：复杂数据类型，存在堆，地址存在栈。

2.函数

当属性的值是一个函数时：

var obj = { foo: function(){} }

同理，js引擎会将函数单独保存在内存中，foo中只保留函数的地址。
也就是说，foo属性的属性描述对象的value属性里，保存的是函数的地址。

{
    foo: {
        [[value]]: 函数的地址
        ...
    }
}

那么：由于函数是一个单独的值，所以它可以在不同的环境（上下文）中执行。比如：

var foo = function(){}
var obj = {foo: foo}
//单独执行
foo()
//obj环境执行
obj.foo()

3.环境变量

js允许在函数体内部，引用当前环境（上下文）的其他变量。
那么由于函数是一个单独的值，可以在不同的环境中执行，那么如何在函数体内部，获得当前的运行环境（上下文）？

this的设计：在函数体内部，指代函数当前的运行环境。比如：

var foo = function(){
    console.log(this.x);
}
var x = 1;
var obj = {
    foo: foo,
    x: 2,
}
//单独执行。这里函数foo在全局环境执行，this.x指向全局环境的x
foo()   //1
//obj环境执行。这里this.x指向obj.x
obj.foo()   //2

因此也就是说，obj.foo是通过obj找到foo，是在obj环境执行。一旦 var foo = obj.foo，变量foo就直接指向函数本身，所以单独foo就是在全局环境。

一些使用时的总结：

对于箭头函数：

箭头函数的this永远指向上级作用域（外层）的this。也就是说，创建箭头函数时，就已经确定了它的this指向。

注：箭头函数不能当作构造函数。比如说new构造和箭头函数同时用时，this指向哪里？报错：
func = () => {}
new func() //throw error

对于非箭头函数：

this具体指向哪个对象，基于运行时函数的执行环境绑定的，而非函数声明时的环境。

this指向全局对象时，在浏览器环境中全局对象是window，在nodejs环境中全局对象是Global。

非严格模式下，如果this指向是undefined或null，转成全局对象。

（1）函数作为对象的方法调用时，this指向该对象。

（2）函数作为普通函数单独调用时，this指向全局对象（非严格模式）。

var obj = {
    foo: function(){
        console.log(this.bar);
    }
    bar: 1;
}
var foo = obj.foo;
var bar = 2;

obj.foo()  //1 this指向obj对象
foo()  //2   this指向全局，浏览器中，全局对象是window

（3）函数作为构造函数调用时，this指向new出来的实例对象。

 function Person(age, name) {
     this.age = age;
     this.name = name
     console.log(this)  // 此处 this 分别指向 Person 的实例对象 p1 p2
 }
var p1 = new Person(18, 'zs')  //this指向p1
var p2 = new Person(18, 'ww')  //this指向p2

（4）通过事件绑定的方法，this指向绑定了事件的对象。

<body>
    <button id="btn">hh</button>
<script>
    var oBtn = document.getElementById("btn");
    oBtn.onclick = function() {
        console.log(this); // this指向绑定了onclick事件的 btn 对象
    }
</script>
</body>

（5）定时器函数，this指向window。

 setInterval(function () {
   console.log(this); // window
 }, 1000);

（6）call、apply、bind动态的改变函数的this的指向。

call和apply：区别：apply传参数数组

var Person = {
    name:"lixue",
    age:21
}
function fn(x,y){
    console.log(x+","+y);
    console.log(this);
}
fn("hh",20);  //this指向全局window
fn.call(Person,"hh",20);  //this指向Person
fn.apply(Person,["hh", 20]);  //this指向Person

bind创建一个新的函数（绑定函数），与被调用函数有相同的函数体，当目标函数被调用时，this绑定到bind()的第一个参数上。比如：

var oDiv1 = document.getElementById("div1");
oDiv1.onclick = function(){
    setTimeout(function(){
        console.log(this);
    }, 1000)
}

此时定时器里的this指向指向window，想要其指向div1，我们可以用bind来改。

var oDiv1 = document.getElementById("div1");
oDiv1.onclick = function(){
    setTimeout(function(){
        console.log(this);
    }.bind(this), 1000)
}

在定时器函数{}外，且在绑定onclick事件内，this指向绑定了事件的div1对象。所以bind(this)将this绑定到这个div1上。

同理用call和apply也可以实现。

不过一般会用这种方法：

var oDiv1 = document.getElementById("div1");
oDiv1.onclick = function(){
    var _this = this;  //把this指向的对象存到变量里
    setTimeout(function(){
        console.log(_this);
    }, 1000);
}
